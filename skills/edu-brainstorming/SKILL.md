# 教育头脑风暴：将教学想法转化为设计方案

通过结构化的协作对话，帮助将教学想法转化为完整的教育设计方案和规格说明。

首先了解当前教学背景和学习者情况，然后一次提出一个问题来完善教学想法。一旦理解了要设计的内容，呈现设计方案并获得教师/用户批准。

<HARD-GATE>
在呈现设计方案并获得用户批准之前，不要调用任何实施 skill、编写任何代码、搭建任何项目或采取任何实施行动。这适用于**每一个**教育项目，无论看起来多么简单。
</HARD-GATE>

## 反模式："这太简单了，不需要设计"

每个教育项目都要经历这个过程。一个测验、一个课堂活动、一份作业布置——所有这些都要。"简单"的教学设计正是那些未经审视的假设造成最多教学浪费的地方。设计可以简短（真正简单的项目几句话即可），但你**必须**呈现设计并获得批准。

## 教育设计检查清单

你必须为以下每一项创建任务并按顺序完成：

1. **分析教学背景** — 检查课程大纲、学生水平、教学目标
2. **了解学习者特征** — 年龄、先验知识、学习风格、特殊需求
3. **明确学习目标** — 知识、技能、情感态度三维目标
4. **提出澄清问题** — 一次一个，理解教学目的/约束/成功标准
5. **提出 2-3 种教学方案** — 附带教学法权衡分析和你推荐的方案
6. **呈现教学设计** — 根据复杂程度分章节，每章后获得用户批准
7. **编写教学设计文档** — 保存到 `docs/edu-designs/YYYY-MM-DD-<topic>-design.md`
8. **设计自检** — 快速检查教学目标对齐、评估有效性、差异化支持
9. **用户审阅设计文档** — 请教师审阅设计文档后再继续
10. **过渡到实施** — 调用 writing-plans skill 创建教学实施计划

## 教育设计流程详解

### 分析教学背景：

- 首先查看当前课程状态（教学大纲、已有资源、学生作业）
- 评估教学范围：如果请求包含多个独立的学习单元（例如"设计一个包含阅读、写作、口语、听力的综合英语课程"），立即标记出来。不要花时间细化需要先分解的项目细节。
- 如果项目太大，帮助教师分解为子单元：哪些是独立的模块、它们如何关联、应该按什么顺序教学？然后通过正常的设计流程对第一个子单元进行头脑风暴。每个子单元都有自己的设计 → 计划 → 实施周期。
- 对于适当规模的项目，一次提出一个问题来完善教学想法
- 尽可能使用选择题，但开放式问题也可以
- 每条消息只提一个问题 — 如果一个话题需要更多探索，拆分为多个问题
- 聚焦于理解：教学目标、学习者约束、成功标准

### 探索教学方案：

- 提出 2-3 种不同的教学方法，附带教学法权衡分析
- 以对话方式呈现选项，包含你的推荐和理由
- 以你推荐的选项开头并解释为什么适合该学习者群体

### 呈现教学设计：

- 一旦你认为理解了要设计的内容，就呈现教学设计
- 根据复杂程度调整每个章节：简单的几句话，复杂的 200-300 字
- 每章后询问到目前为止是否正确
- 涵盖：教学目标、学习者分析、教学策略、评估方法、差异化支持
- 准备好返回并澄清如果某些内容不合理

### 教学设计原则：

- 将学习体验分解为更小、可管理的单元，每个单元有明确的学习目标
- 对于每个单元，你应该能够回答：学生将学会什么、如何评估、需要什么先验知识？
- 有人能在不阅读内部设计的情况下理解一个单元的学习目标吗？你能调整教学方法而不影响整体课程结构吗？如果不能，边界需要调整。
- 更小、边界清晰的单元也更容易让教师实施 — 你能更好地推理可以一次性在课堂中完成的教学，当活动聚焦时你的教学更可靠。当一个课时包含太多内容时，这通常是它超载的信号。

### 在现有课程中工作：

- 在提出新设计之前探索当前课程结构。遵循现有教学模式和评估体系。
- 当现有课程存在影响工作的问题时（例如某个单元内容过多、目标不清晰、活动纠缠），将针对性改进作为设计的一部分 — 就像优秀教师在正在教授的课程中进行改进一样。
- 不要提出无关的课程重构。专注于服务于当前教学目标的内容。

## 设计完成后

### 文档：

- 将经过验证的教学设计写入 `docs/edu-designs/YYYY-MM-DD-<topic>-design.md`
  -（用户对设计文档位置的偏好优先于此默认位置）
- 如果可用，使用 elements-of-style:writing-clearly-and-concisely skill
- 将设计文档提交到 git

### 设计自检：

编写设计文档后，用新的眼光审视它：

1. **教学目标对齐扫描**：学习目标、教学活动和评估方法是否一致？
2. **内部一致性**：各章节之间有矛盾吗？教学策略与学习目标匹配吗？
3. **范围检查**：这足够聚焦以放入单个实施计划，还是需要分解为多个课时？
4. **差异化支持检查**：是否为不同学习风格和能力水平的学生提供了支持？
5. **评估有效性**：评估方法能否有效测量学习目标的达成？

内联修复任何问题。不需要重新审阅 — 直接修复并继续。

### 用户审阅关卡：

设计审阅循环通过后，请教师在继续之前审阅书面设计：

> "教学设计文档已编写并保存到 `<path>`。请审阅它，如果需要在开始编写教学实施计划之前做任何更改，请告诉我。"

等待教师的回复。如果他们请求更改，进行更改并重新运行设计审阅循环。只有在教师批准后才能继续。

### 实施：

- 调用 writing-plans skill 创建详细的教学实施计划
- 不要调用任何其他 skill。writing-plans 是下一步。

## 核心原则

- **一次一个问题** — 不要用多个问题压倒教师
- **优先选择题** — 可能比开放式问题更容易回答
- **无情地聚焦学习目标** — 从所有设计中移除与目标无关的活动
- **探索替代教学法** — 在确定之前始终提出 2-3 种方案
- **增量验证** — 呈现设计，在继续之前获得批准
- **保持灵活** — 当某些内容不合理时返回并澄清
- **以学习者为中心** — 所有设计决策都应考虑学习者的需求和特征

## 视觉辅助

一个基于浏览器的辅助工具，用于在头脑风暴期间展示教学设计模型、课程结构图和视觉选项。作为工具可用 — 不是模式。接受辅助意味着它可用于受益于视觉处理的问题；这并不意味着每个问题都通过浏览器。

**提供辅助：** 当你预期即将到来的问题将涉及视觉内容（课程流程图、活动布局、评估量规）时，一次性征求同意：

> "我们正在处理的一些教学设计内容如果能在网页浏览器中展示可能会更容易理解。我可以随时整理课程结构图、活动流程、对比图和其他视觉辅助。这个功能仍然很新，可能会消耗较多 token。想试试吗？（需要打开本地 URL）"

**这个提议必须是单独的消息。** 不要与澄清问题、背景摘要或任何其他内容合并。消息应该只包含上述提议，不含其他内容。等待教师的回复后再继续。如果他们拒绝，继续进行纯文本头脑风暴。

**每个问题的决策：** 即使用户接受了，也要为**每个问题**决定是使用浏览器还是终端。测试标准：**教师通过看比通过读能更好地理解这个吗？**

- **使用浏览器**处理视觉内容 — 课程流程图、单元结构图、活动布局对比、评估量规设计、并排教学方案对比
- **使用终端**处理文本内容 — 教学目标问题、教学法选择、权衡列表、A/B/C/D 文本选项、范围决策

一个关于教学活动的问题不自动就是视觉问题。"在这种情况下，合作学习是什么意思？"是概念问题 — 使用终端。"哪种小组活动布局效果更好？"是视觉问题 — 使用浏览器。

如果他们同意使用辅助，在继续之前阅读详细指南：
`skills/edu-brainstorming/visual-companion.md`

## 教育场景快速参考

| 教育场景 | 关键考虑因素 |
|---------|-------------|
| 课程设计 | 学习目标、学习者分析、教学序列、评估策略 |
| 测验/考试 | 目标对齐、难度分布、题型多样性、评分标准 |
| 课堂活动 | 参与度、协作机会、时间控制、材料准备 |
| 差异化教学 | 学习者差异、支持策略、扩展活动、补救措施 |
| 项目式学习 | 驱动问题、探究过程、成果展示、反思环节 |
