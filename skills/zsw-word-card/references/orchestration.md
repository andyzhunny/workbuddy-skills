# zsw-word-card orchestration

## 1. 角色定位

`zsw-word-card` 不是新的词典，也不是新的制图引擎。
它只做一件事：把 `ljg-word` 和 `ljg-card` 编排成一条稳定工作流。

## 2. Source of Truth

`ljg-word` 的完整输出是唯一可信内容源。

执行卡片步骤时：
- 可以为了版式做 HTML 结构重排
- 可以调整标题区、列表样式、引用样式
- 可以压缩空白和视觉节奏

但**不允许**：
- 先写一个更短的摘要再交给 `ljg-card`
- 只保留原始画面、核心意象或金句中的一部分
- 删除关键例子、关键人物、关键论证
- 用记忆重写第一步内容，导致卡片和词解不一致

原则：**卡片是完整词解的视觉化，不是完整词解的缩水版。**

## 3. 输入处理

- 输入可以是中文词、英文词、术语、概念词
- 若词条是中文且存在常见英文对应，可在标题中补出英文
- 若词条是英文，应补出中文释义与音标
- 默认按“单个词条”处理；若用户一次给多个词，提醒逐个生成更稳

## 4. 模具选择

优先级不是固定，而是内容驱动：

### 首选 `-l`
适用于：
- 大多数抽象概念词
- 解释以连续阅读为主
- 内容张力来自语言本身，而非图表结构

### 使用 `-i`
适用于：
- 词解内部有强对照、强结构、实验、机制、分层模型
- 存在“X + Y = Z”式关系，适合图解
- 有典型案例可被转成流程或框图

### 使用 `-v`
适用于：
- 词解更像思维网络而不是线性文章
- 词源、隐喻、应用场景之间有多跳联想
- 更适合视觉笔记式组织

### 其他模具
`-m`、`-c`、`-w` 默认不优先；只有当内容明确要求多页拆分、漫画叙事或白板框图时再用。

## 5. 输出协议

最终交付必须包含两部分，顺序固定：

### A. 对话中的完整词解
直接输出 `ljg-word` 的完整 Markdown 结果。

### B. 卡片结果
- 简要说明选用了哪种模具，以及为什么
- 展示生成好的 PNG
- 卡片内容必须与 A 段保持语义一致

## 6. 自检清单

生成结果前快速核对：

- [ ] 是否真的先完成了 `ljg-word` 的完整词解？
- [ ] 给 `ljg-card` 的是否是完整词解而非摘要？
- [ ] 最终回复里是否先给了完整词解？
- [ ] 模具选择是否有内容依据，而不是机械默认？
- [ ] 卡片是否保留了词解中的核心论点、关键例子和金句？
